> 阶段范围：项目初始化 → v0.1.0（核心域模型与基础接口完成）

## 一、本阶段目标与完成情况

本阶段的目标是搭建 **SmartQA** 后端骨架，打通「用户 → 文档 → 问答」三条主流程，为后续 RAG、微服务演进打基础。


| 任务项                  | 状态  | 说明                        |
| -------------------- | --- | ------------------------- |
| Spring Boot 3 项目初始化  | ✅   | Java 21 + Maven Wrapper   |
| PostgreSQL + JPA 数据层 | ✅   | 四张核心表自动建表                 |
| 用户注册 / 登录            | ✅   | BCrypt 加密，占位 Token        |
| 文档上传落盘               | ✅   | 本地 `uploads` 目录，元数据入库     |
| 问答社区 CRUD            | ✅   | 问题列表 / 详情 / 回答            |
| API 文档（Swagger）      | ✅   | springdoc-openapi         |
| 项目 README 与目录整理      | ✅   | 代码从 `smartqa/` 子目录移至仓库根目录 |


**本阶段交付物：** 可独立启动的后端服务，通过 Postman 或 Swagger UI 完成端到端接口验证。

## 二、本阶段学会了什么

### 1. Spring Boot 3 工程化起步

- 使用 **Spring Initializr** 思路创建项目，理解 `pom.xml` 中各 Starter 的职责：
  - `spring-boot-starter-web`：REST API
  - `spring-boot-starter-data-jpa`：ORM 与 Repository
  - `spring-boot-starter-validation`：`@Valid` 参数校验
- 掌握 `application.yml` 配置数据源、JPA、`multipart` 上传大小等常用项。
- 使用项目自带的 `mvnw` / `mvnw.cmd`，减少本机 Maven 版本差异带来的问题。

### 2. JPA 实体建模与关系映射

- 完成四张业务表对应的实体：`User`、`Document`、`Question`、`Answer`。
- 理解 `@ManyToOne`、`@JoinColumn`、`FetchType.LAZY` 在关联查询中的含义。
- 使用 `@CreationTimestamp`、`@UpdateTimestamp` 自动维护时间字段。
- 通过 `ddl-auto: update` 让 Hibernate 在开发阶段自动同步表结构（适合学习，生产需改用 Flyway/Liquibase）。

### 3. 分层架构的实践

形成了清晰的分层习惯：

```
Controller（接口 + 校验） → Service（业务逻辑） → Repository（数据访问） → Entity（领域模型）
```

- **DTO** 与 **Entity** 分离：注册、登录、创建问题等请求体单独定义，避免直接把实体暴露给 API。
- **Repository** 继承 `JpaRepository`，并在需要时写 `@Query` + `JOIN FETCH` 解决懒加载问题。

### 4. 用户与安全相关基础

- 仅引入 `spring-security-crypto`，用 **BCrypt** 对密码哈希存储，不引入完整 Spring Security 过滤器链（降低首阶段复杂度）。
- 登录返回 **UUID 占位 Token**，明确与后续 JWT 方案的差距。
- 在 `User` 实体上对 `password` 使用 `@JsonIgnore`，避免注册接口响应泄露密码哈希。

### 5. 文件上传与本地存储

- `MultipartFile` 接收文件，`Files.copy` 写入磁盘。
- 用 **UUID + 原扩展名** 作为存储文件名，避免重名覆盖。
- 上传目录通过 `@Value("${smartqa.upload-dir}")` 配置，启动时 `Files.createDirectories` 确保目录存在。
- 文档元数据（标题、路径、状态 `PROCESSING`）写入数据库，为后续解析与向量化预留字段。

### 6. 问答模块与 JPA 查询技巧

- 问题、回答的创建流程：先校验 `userId` 是否存在，再设置 `@ManyToOne` 关联用户。
- 列表 / 详情查询遇到 **LazyInitializationException** 或 JSON 序列化关联对象为空时，学会：
  - 在 Repository 用 `JOIN FETCH` 一次性加载 `askedBy`
  - 在实体上加 `@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})`
- 在 Service 方法上使用 `@Transactional` / `@Transactional(readOnly = true)` 控制事务边界。

### 7. API 文档与联调

- 集成 **springdoc-openapi**，通过 Swagger UI 在线调试接口。
- 熟悉 REST 约定：`POST` 创建、`GET` 查询，统一前缀 `/api/users`、`/api/documents`、`/api/questions`。

### 8. Git 与项目演进意识

- 按功能拆分 commit（实体 → 用户 → 文档 → 问答 → 文档工具）。
- 打 Tag `v0.1.0-核心域模型与基础接口完成`，与 README 中的版本路线图对齐。
- 将后端代码从 `smartqa/` 子目录提升到仓库根目录，便于后续与 `frontend/`、`notes/` 并列管理。

### 9. AI 辅助开发（Cursor）

- 本阶段在 Cursor 中借助 **Claude** 完成用户注册登录首版；完整协作记录见 **[第六节](#六使用-claudecursor辅助开发记录)**。

## 三、遇到的问题与解决思路

### 1. PostgreSQL 连接失败

**现象：** 启动报 `Connection refused` 或认证失败。

**原因：** 数据库未启动、库名 `smartqa` 未创建、用户名密码与 `application.yml` 不一致。

**处理：**

```sql
CREATE DATABASE smartqa;
```

本地修改 `spring.datasource.username` / `password`，确认 PostgreSQL 服务已启动。

---

### 2. JPA 懒加载导致接口报错或 JSON 异常

**现象：** 查询问题列表时，关联的 `askedBy` 为 `null`，或控制台出现 `LazyInitializationException`。

**原因：** `@ManyToOne(fetch = LAZY)` 在 Session 关闭后访问关联对象。

**处理：** 在 `QuestionRepository` 中编写 `JOIN FETCH` 查询；读操作 Service 加 `@Transactional(readOnly = true)`；实体补充 Jackson 忽略 Hibernate 代理字段的配置。

---

### 3. 注册接口响应里出现 password 字段

**现象：** 返回的 User JSON 中包含加密后的密码。

**处理：** 在 `User.password` 上添加 `@JsonIgnore`，只用于持久化，不参与序列化。

---

### 4. 文档上传在 Postman 中失败

**现象：** 400 或文件为空。

**原因：** 未使用 `multipart/form-data`，或字段名与接口不一致。

**处理：** Postman 选择 form-data，字段名为 `file`（文件类型）、`title`（文本）、`userId`（文本/数字）；注意单文件上限 50MB 的配置。

---

### 5. 登录 Token 无法用于后续接口鉴权

**现象：** 登录拿到 `token`，但其他接口并未校验。

**原因：** 当前仅为 `UUID.randomUUID()` 占位，未接入 JWT，也未配置拦截器。

**处理：** 属预期内的 v0.1 限制；V1.0 计划引入 Spring Security + JWT。本阶段接口联调仍通过显式传 `userId` 完成。

---

### 6. 业务异常直接返回 500

**现象：** 用户名重复、用户不存在等抛出 `IllegalArgumentException`，客户端收到 500 而非友好 JSON。

**原因：** 尚未实现 `@ControllerAdvice` 全局异常处理。

**处理：** 本阶段先保证主流程跑通；下一版本统一封装 `Result` 与错误码（已列入 V1.0 规划）。

---

### 7. 项目目录结构调整

**现象：** 初期代码放在 `smartqa/` 子模块下，与 README 中「仓库根目录即后端」的描述不一致。

**处理：** 提交「移动代码位置」，将 `pom.xml`、`src/` 等移至仓库根目录；后续启动命令统一为在根目录执行 `.\mvnw spring-boot:run`（README 中部分 `cd smartqa` 表述需在后续同步修正）。

---

### 8. 开发环境与工具链

- **JDK 版本：** 需与 `pom.xml` 中 `java.version`（21）一致，避免编译或运行字节码版本不匹配。
- **PowerShell：** 链式命令不能用 `&&`，应使用 `;` 分隔。
- **敏感配置：** 数据库密码写在 yml 中仅适合本地开发，勿提交生产密钥（`.gitignore` 可考虑本地 `application-local.yml`）。

## 四、当前版本的已知限制（备忘）

以下限制已在 README「已知限制」中记录，作为下一阶段迭代的输入：


| 限制                     | 影响             |
| ---------------------- | -------------- |
| Token 未鉴权              | 接口无法识别「当前登录用户」 |
| 文档 `status=PROCESSING` | 无解析、无向量化       |
| 无全局异常处理                | 错误响应不统一        |
| 无前端                    | 仅 API 联调       |
| 无单元 / 集成测试覆盖           | 重构风险较高         |

## 五、下一阶段计划（V1.0 方向预览）

结合项目路线图，建议下一阶段优先：

1. **JWT 鉴权**：替换占位 Token，保护需登录的接口。

4. **补充接口测试**：至少覆盖用户注册、文档上传、问答主路径。
5. **同步 README**：启动路径、目录结构与真实仓库一致。

## 六、使用 Claude（Cursor）辅助开发记录

> 供后续开发者参考：如何在 SmartQA 项目中**有节制地**使用 AI 写代码，而不是 blindly copy。  
> 对应 Git 提交：`a9e9b0b` — `Cursor上使用Claude实现用户的注册与登录`

### 6.1 工具与环境


| 项     | 说明                                          |
| ----- | ------------------------------------------- |
| IDE   | [Cursor](https://cursor.com/)               |
| 模型    | Claude（DeepSeek v4 Pro）                     |
| 项目上下文 | 打开仓库根目录，确保 AI 能索引 `entity/`、`pom.xml` 等已有代码 |
| 前置条件  | 项目框架搭建成功，可运行。                               |


### 6.2 本阶段 AI 参与范围


| 模块         | 是否使用 Claude | 说明                    |
| ---------- | ----------- | --------------------- |
| 用户注册 / 登录  | ✅ **是**     | 首版由 Claude 生成，人工验证后合入 |
| 实体类、数据库配置  | ❌ 否         | 手动完成，为 AI 提供上下文       |
| 文档上传       | ✅ **是**     | Claude 生成             |
| 问答 CRUD    | ✅ **是**     | Claude 生成             |
| Swagger 集成 | ❌ 否         | 自行查阅文档添加依赖            |


**结论：** v0.1 中 AI 主要用于**破冰**——在已有实体与 Repository 的前提下，快速搭出第一个完整垂直切片（Controller → Service → DTO）。

### 6.3 推荐协作流程

```
1. 自己完成领域建模（Entity + Repository）
       ↓
2. 用自然语言向 Claude 描述 API 契约与约束
       ↓
3. AI 生成分层代码 → 本地 compile / 启动
       ↓
4. Swagger 或 Postman 联调，发现问题
       ↓
5. 人工 Code Review：安全、分层、异常、是否过度设计
       ↓
6. 小步提交，commit message 注明是否经 AI 辅助
```

### 6.4 实际使用的 Prompt 示例（可复用）

以下 Prompt 根据提交 `a9e9b0b` 反推整理，后续开发者可按需改写。

**Prompt 1：用户注册与登录（首版）**

```text
项目：Spring Boot 3 + JPA，已有 User 实体和 UserRepository。

请实现用户模块：
1. POST /api/users/register
   - 请求体：username、password、email（RegisterRequest）
   - 校验用户名唯一；密码用 BCrypt 加密后入库
2. POST /api/users/login
   - 请求体：username、password（LoginRequest）
   - 校验密码；成功返回 { token, username }
   - token 先用 UUID 占位，后续再换 JWT

要求：
- 分层：UserController、UserService、DTO
- 仅引入 spring-security-crypto，不要完整 Spring Security 过滤器链
- Controller 使用 @Valid
- 业务错误抛 IllegalArgumentException
```

**Prompt 2：生成后追问（建议必做）**

```text
请检查：
1. 注册接口响应是否会泄露 password 字段？需要的话加 @JsonIgnore
2. BCryptPasswordEncoder 是否应注册为 @Bean？
3. 与现有 User 实体、表名 t_user 是否一致？
```

**Prompt 3：联调失败时（示例）**

```text
注册接口返回 500，日志显示 ...（粘贴关键栈）
当前 User 实体和 application.yml 如下：（@文件 或粘贴片段）
请帮我定位原因并给出最小改动。
```

### 6.5 Claude 生成的文件清单（提交 a9e9b0b）


| 文件                               | 作用                                     |
| -------------------------------- | -------------------------------------- |
| `config/SecurityConfig.java`     | 注册 `BCryptPasswordEncoder` Bean        |
| `controller/UserController.java` | `register`、`login` 两个端点                |
| `dto/RegisterRequest.java`       | 注册请求体 + `@NotBlank`                    |
| `dto/LoginRequest.java`          | 登录请求体                                  |
| `service/UserService.java`       | 注册唯一性校验、BCrypt、UUID Token              |
| `repository/UserRepository.java` | 补充 `findByUsername`、`existsByUsername` |
| `pom.xml`                        | 增加 `spring-security-crypto` 依赖         |


生成后由开发者本地启动验证，并在后续提交中自行补充 `@JsonIgnore`（密码不序列化）等修正。

### 6.6 生成代码中值得学习的点

```java
// UserService：注册流程清晰，适合作为后续模块模板
if (userRepository.existsByUsername(request.getUsername())) {
    throw new IllegalArgumentException("用户名已存在: " + request.getUsername());
}
user.setPassword(passwordEncoder.encode(request.getPassword()));
```

```java
// SecurityConfig：只引入加密能力，不引入完整 Security 链，符合 v0.1 降复杂度策略
@Bean
public BCryptPasswordEncoder bCryptPasswordEncoder() {
    return new BCryptPasswordEncoder();
}
```

### 6.7 必须人工核对的事项（Checklist）

在合并 AI 代码前，建议逐项勾选：

- **编译通过**：`.\mvnw compile`
- **启动无报错**：数据库已创建、配置正确
- **Swagger 可调通**：注册 → 登录顺序验证
- **密码不出现在 JSON 响应**：`User.password` 加 `@JsonIgnore`
- **依赖最小化**：未误引入 `spring-boot-starter-security` 全量包
- **与实体字段一致**：DTO 字段名、类型与 `User` 对齐
- **无硬编码密钥**：密码、Token 逻辑符合当前阶段设计
- **理解每一行**：能向他人解释 Service 中事务、加密、异常的含义

### 6.8 使用 AI 时踩过的坑与经验


| 现象                   | 原因                   | 处理方式                           |
| -------------------- | -------------------- | ------------------------------ |
| 注册响应带 `password` 哈希  | 实体默认 JSON 序列化所有字段    | 人工加 `@JsonIgnore`              |
| 引入完整 Spring Security | Prompt 未约束「仅 crypto」 | Prompt 中明确「不要过滤器链」             |
| 与现有包名 / 表名不一致        | AI 未读到最新 Entity      | 用 `@` 引用文件或粘贴实体代码              |
| 一次生成过多文件难以 Review    | Prompt 范围过大          | 按「一个垂直切片」拆分 Prompt             |
| 直接提交未测试代码            | 过度信任 AI              | 坚持「生成 → 编译 → 联调 → Review → 提交」 |


### 6.9 何时适合 / 不适合使用 Claude


| 适合 ✅               | 不适合 ❌                    |
| ------------------ | ------------------------ |
| 重复性分层代码（CRUD 骨架）   | 核心业务规则未想清楚时              |
| 根据报错栈快速定位问题        | 不读代码直接上线                 |
| 解释 JPA / Spring 概念 | 让 AI 决定整体架构而不加约束         |
| 编写 README、设计文档初稿   | 将数据库密码、API Key 贴进 Prompt |


### 6.10 给后续开发者的建议

1. **先建模、后生成**：Entity / 表结构由人定，AI 填 Controller / Service。
2. **Prompt 写清约束**：技术栈版本、不要引入的依赖、API 路径、错误处理方式。
3. **commit 可追溯**：像 `a9e9b0b` 一样在 message 中注明「Cursor + Claude」，便于日后 blame。
4. **文档与代码同步**：本阶段总结、 `docs/design-v1.md` 应在版本完成后更新。
5. **v0.1 之后**：问答、文档模块已证明可自行实现；V1.0 的 JWT、异步解析等仍可用 AI 辅助，但**安全与并发相关代码必须人工 Review**。

## 七、个人反思

本阶段的最大收获不是「写完了几个接口」，而是建立起 **从领域模型到可运行 API** 的完整闭环：会建表、会分层、会联调、会用 Git 标记里程碑。借助 Claude 完成用户模块首版后，文档上传与问答模块改为自行实现，对分层的理解明显加深。遇到的懒加载、密码序列化、文件上传格式等问题，都是后端开发中的典型坑，提前踩过对后续接入 AI/RAG 很有帮助。

SmartQA 的长期目标是企业级 RAG 问答平台；v0.1 虽然还没有「智能」，但 **用户、文档、问答** 三条业务线已经就位，下一阶段可以在稳固的工程化基础上，逐步补上安全、解析与可观测性。
